# 团队工作流程

## 团队管理目标

所有流程设计服务于四个目标：
1. **快速迭代** — 产品功能快速开发上线
2. **高效协同** — 信息高效同步，减少沟通损耗
3. **自动运转** — 团队自主运转，创始人只做方向决策
4. **持续成长** — 经验沉淀为规则，能力持续提升

---

## 流程总览

```
[1] 需求输入 → [2] 方案设计 → [3] 技术评估 → [4] 开发实现 → [5] 测试验证 → [6] Code Review → [7] 本地验收 → [8] 部署上线 → [9] 线上验证 → [10] 沉淀归档
```

每一步都有明确的**输入、输出、验证条件和异常处理**。只有当前步骤的输出满足下一步的输入要求时，才能往下走。

---

## [1] 需求输入

### 触发条件
创始人提出需求（口头描述、方向性想法、问题反馈）

### 输入
- 创始人的描述或指令
- 相关上下文（如果有）

### 执行者
Project Lead

### 过程
1. 理解创始人意图（不确定时复述确认，不自作主张）
2. 判断任务类型：
   - **新功能** → 进入 [2] 方案设计
   - **Bug 修复** → 直接进入 [4] 开发实现（快速通道）
   - **方向性探索** → Product Designer 调研后汇报创始人

### 输出
- 明确的任务描述（给 Product Designer 或 Dev）
- 任务类型标注（新功能 / 小优化 / Bug / 探索）

### 异常处理
| 异常 | 处理 |
|------|------|
| 需求不清晰 | Project Lead 向创始人提问澄清，不猜测 |
| 需求涉及架构变更 | 先让 CTO 评估可行性，再决定是否推进 |
| 需求与已有决策冲突 | 查 decisions.md 确认，冲突时报告创始人 |

---

## [2] 方案设计

### 触发条件
[1] 输出明确的新功能任务

### 输入
- 任务描述
- 相关的现有需求文档和规格文档
- decisions.md（已有决策 + 产品愿景）
- board.md（优先级上下文）

### 执行者
Product Designer

### 过程
1. 读取相关文档，理解现有系统
2. 产出需求文档：产品逻辑 + 交互行为描述 + 文案 + 信息架构 + **影响范围分析**
3. 产出 .pen 设计稿（遵守 design-system.md）
4. 需求文档中不写视觉实现数值（px、颜色、CSS）
5. .pen 中不写业务逻辑

### 输出
- 需求文档：`docs/product/requirements/[需求] {slug}.md`（status: review）
- .pen 设计稿更新（如涉及 UI 变更）
- design-system.md 更新（如有新组件或规范变更）

### 验证条件
- [ ] 需求文档包含影响范围分析
- [ ] 需求文档零视觉实现数值
- [ ] .pen 与 design-system.md 一致
- [ ] .pen 已 get_screenshot 验证

### 异常处理
| 异常 | 处理 |
|------|------|
| 需求涉及多个模块 | 影响范围分析中列出所有受影响模块，CTO 评估关联影响 |
| 与现有功能冲突 | 在需求文档中标注冲突点，创始人决定取舍 |
| 设计需要突破 design-system.md 规范 | 先更新 design-system.md，再更新 .pen |

### 创始人介入点
- 新功能方案需创始人确认后才能进入开发
- Bug 修复跳过此步，不需要确认

---

## [3] 技术评估

### 触发条件
[2] 输出需求文档（status: review）

### 输入
- 需求文档
- .pen 设计稿
- 现有代码和架构

### 执行者
CTO

### 过程
1. 评估技术可行性
2. **完整的影响范围分析**：受影响的代码文件、配置文件、定时任务、数据库、基础设施、已有数据
3. 识别技术风险和已知限制
4. 产出 Dev 实施步骤清单（具体到文件和函数级别）
5. 估算实施复杂度

### 输出
- 技术方案文档：`docs/technical/deliverables/[方案] {slug} {date}.md`（status: review）
- 包含：影响范围清单 + 实施步骤 + 风险评估

### 验证条件
- [ ] 影响范围分析覆盖：代码、配置、定时任务、数据库、基础设施
- [ ] 已识别与现有系统的兼容性问题
- [ ] 实施步骤具体到文件级别

### 异常处理
| 异常 | 处理 |
|------|------|
| 技术不可行 | 反馈 Product Designer 调整方案范围 |
| 需要架构变更 | 报告创始人确认 |
| 已知限制影响产品体验 | 反馈 Product Designer，协商替代方案 |
| 遗漏影响范围 | 这是严重失误，发现后立即补充，写入经验教训 |

---

## [4] 开发实现

### 触发条件
- 新功能：[3] 技术方案确认
- Bug 修复：[1] 直接触发

### 输入
- 需求文档（产品逻辑 + 交互行为）
- .pen 设计稿（视觉实现）
- design-system.md（设计约束）
- 技术方案文档（实施步骤）

### 执行者
Dev

### 过程
1. 读取所有输入文档
2. 读取 `.claude/rules/backend.md` 或 `frontend.md`（已知的坑）
3. 读取相关 Model 文件（引用字段前必读）
4. 按技术方案步骤实现
5. **本地验证**（必须全部通过才能交付）

### 输出
- 代码变更（本地，不推送）
- 验证结果报告

### 验证条件（Quality Gates）

#### 前端
```bash
cd frontend && npm run build        # 编译通过
cd frontend && npx tsc --noEmit     # 类型检查通过
```

#### 后端
```bash
cd backend && uv run python -c "from app.main import app"   # import 通过
```

#### Celery 任务（如涉及）
- 新任务已在 `tasks/__init__.py` 注册
- async session 使用模式正确（参考 embedding_backfill.py）

**未通过验证不得报告完成。**

### 异常处理
| 异常 | 处理 |
|------|------|
| 技术方案步骤不明确 | 读代码自行理解，不盲目猜测。实在不确定时问 CTO |
| 实现中发现方案有问题 | 停止实现，反馈 CTO 评估调整 |
| 构建/类型检查失败 | 自行修复直到通过，不带着错误交付 |
| Model 字段不存在 | 确认是否需要 migration，反馈 CTO |

---

## [5] 测试验证

### 触发条件
[4] Dev 报告开发完成 + 验证通过

### 输入
- 需求文档（验证需求覆盖度）
- Dev 的代码变更
- 技术方案（了解实施范围）

### 执行者
Tester（使用 Codex 交叉验证）

### 过程
1. 验证 Codex CLI 可用（`which codex && codex --version`）
2. 对照需求文档逐项验证功能实现
3. 测试边界条件和异常场景
4. API 端点验证（curl）
5. 使用 Codex 从不同 AI 视角审查代码逻辑
6. 产出测试报告

### 输出
- 测试报告：按需求条目逐项 PASS / FAIL / NOT_TESTED
- 问题列表（分级：Blocker / Major / Minor / Suggestion）
- 结论：通过 / 有条件通过 / 不通过

### 验证条件
- [ ] 需求文档中每一条功能点都已测试或标注 NOT_TESTED（附原因）
- [ ] 所有 Blocker 和 Major 问题有复现步骤

### 异常处理
| 异常 | 处理 |
|------|------|
| 发现 Blocker | 返回 [4]，Dev 修复后重新测试 |
| 发现 Major | 返回 [4]，Dev 修复后重新测试 |
| 发现 Minor | 记录在报告中，不阻塞流程，可在后续迭代修复 |
| Codex 不可用 | 通知 Project Lead，不阻塞测试，但标注"未进行交叉验证" |
| 需求文档不清晰无法判断 PASS/FAIL | 标注 NOT_TESTED + 原因，反馈 Product Designer 补充需求 |

---

## [6] Code Review

### 触发条件
[5] 测试报告结论为"通过"或"有条件通过"（Blocker/Major 已修复）

### 输入
- Dev 的代码变更
- 测试报告
- 技术方案

### 执行者
CTO

### 过程
1. 审查代码安全性（注入、泄漏、权限）
2. 审查性能（N+1 查询、大数据量、内存）
3. 审查与现有代码一致性
4. Model 字段交叉验证
5. 确认影响范围内的所有变更已处理

### 输出
- 审查结果：Critical / High / Medium / Low 问题列表
- 结论：可合并 / 需修复后合并

### 验证条件
- [ ] 零 Critical 问题
- [ ] 零 High 问题（或已有修复方案）

### 异常处理
| 异常 | 处理 |
|------|------|
| 发现 Critical | 返回 [4]，Dev 修复。CTO 可直接修复 < 10 行的简单问题 |
| 发现 High | 返回 [4]，Dev 修复 |
| 发现 Medium/Low | 记录，不阻塞，后续迭代处理 |
| 发现方案级别的问题 | 返回 [3]，CTO 重新评估方案 |

---

## [7] 本地验收

### 触发条件
[6] CTO 审查通过（零 Critical/High）

### 输入
- 完整的代码变更
- 测试报告
- CTO 审查报告

### 执行者
Project Lead 协调，创始人验收

### 过程
1. 本地 `docker compose build && docker compose up` 运行完整环境
2. 创始人查看本地效果
3. 对照需求文档确认功能符合预期

### 输出
- 验收结论：通过 / 不通过（附反馈）

### 异常处理
| 异常 | 处理 |
|------|------|
| 创始人不满意 | 明确反馈点，判断返回哪一步（[2] 方案问题 / [4] 实现问题） |
| 本地环境跑不起来 | CTO 排查环境问题 |
| 功能正确但视觉不对 | 返回 [2]，Product Designer 调整 .pen |

### 创始人介入点
- 这是创始人验收功能的唯一环节
- 验收在本地环境，不在线上

---

## [8] 部署上线

### 触发条件
[7] 创始人验收通过

### 输入
- 验收通过的代码
- CTO 审查通过的技术方案

### 执行者
CTO

### 过程
1. 确认所有本地验证已通过
2. 确认变更影响范围内的配置/migration 已准备
3. 提交代码到 main
4. 推送触发 GitHub Actions 部署（或手动部署）
5. 等待部署完成

### 输出
- 代码已推送到 main
- 部署已完成

### 验证条件
- [ ] git push 成功
- [ ] GitHub Actions 部署成功

### 异常处理
| 异常 | 处理 |
|------|------|
| 部署失败 | CTO 排查原因并修复，不反复试错式 push |
| 需要 migration | 部署后执行 alembic upgrade head |
| 需要配置变更 | 确认 .env 已更新 |

---

## [9] 线上验证

### 触发条件
[8] 部署完成

### 输入
- 部署成功的确认
- 需求文档（验证功能）

### 执行者
CTO

### 过程
1. API smoke test（不只是 200，要验证返回内容）
2. **Worker 日志检查**（确认无报错，尤其是 Celery 任务）
3. 前端页面访问验证
4. 相关定时任务确认正常
5. 影响范围内的其他服务确认未受影响

### 输出
- 线上验证报告：所有检查项 PASS / FAIL
- 确认：线上功能正常

### 异常处理
| 异常 | 处理 |
|------|------|
| API 异常 | CTO 排查日志，定位问题 |
| Worker 报错 | CTO 检查 Celery 日志，必要时 Dev 修复 |
| 服务受影响 | CTO 评估是否回滚（git revert）还是热修 |
| 数据异常 | CTO 检查 migration 是否正确执行 |

---

## [10] 沉淀归档

### 触发条件
[9] 线上验证通过

### 输入
- 上线成功的功能
- 开发过程中的经验教训

### 执行者
Project Lead 协调各角色

### 过程
1. **Product Designer** 将 [需求] 细节原样合并进 [规格]（`docs/product/modules/[规格] {slug}.md`），合并后归档 [需求]
2. **CTO** 将技术经验写入 `.claude/rules/`（自动加载给未来的 Agent）
3. **Project Lead** 更新 `docs/board.md`
4. 需求文档 status 更新为 `implemented`
5. 如有新的踩坑经验，写入 `memory/` 对应的角色经验文件

### 输出
- 规格文档已更新
- 经验已写入规则文件
- 看板和变更记录已更新

### 验证条件
- [ ] 规格文档反映线上最新状态
- [ ] 新发现的技术坑已写入 `.claude/rules/`
- [ ] board.md 已更新

## 快速通道（Bug 修复 / 小改动）

```
[1] 需求输入 → [4] Dev 修复 + 本地验证 → [6] CTO Review → [8] 部署 → [9] 线上验证
```

跳过 [2] 方案设计、[3] 技术评估、[5] 测试、[7] 本地验收。

适用条件：
- 明确的 Bug（不是功能变更）
- 变更范围小（< 50 行代码）
- 不涉及架构或数据变更

```
 ↑
[7] 本地验收 ──→ 不通过 → 判断原因 → 返回 [2] 或 [4]
 ↑
[6] Code Review ──→ Critical/High → 返回 [4] Dev 修复
 ↑
[5] 测试 ──→ Blocker/Major → 返回 [4] Dev 修复
 ↑
[4] 开发 ──→ 方案问题 → 返回 [3] CTO 调整
 ↑
[3] 技术评估 ──→ 不可行 → 返回 [2] 调整方案
 ↑
[2] 方案设计 ──→ 创始人不同意 → 返回 [1] 重新理解需求
 ↑

```

## 关键约束

### 信息传递
- Agent 间传**文件路径**，不复制内容
- 任务指令包含：背景 + 输入文件路径 + 期望输出 + 约束 + 验证方式

### 并行规则
- 最多 6 个 Agent 同时工作
- 无依赖的任务可并行
- 有依赖的任务必须等前置完成
- 多个 Agent 不得同时编辑同一文件
- 同一文件不同区域 → Worktree 隔离

### 创始人介入点（仅两处）
1. **[2] 方案设计后** — 确认方向
2. **[7] 本地验收** — 确认功能符合预期

### 部署纪律
- 本地验证通过后才推送
- 不允许"改一行推一次" — 统一部署
- 部署和运维操作分开

## 任务管理

### 双层系统

| 层次 | 载体 | 用途 |
|------|------|------|
| 战略层 | board.md（git） | 跨会话任务全景 |
| 战术层 | Claude Code TaskCreate | 会话内 Agent 协调 |

会话开始读 board.md 恢复上下文，会话结束同步回 board.md。

### 任务分配
- 首次由 Lead 指派（spawn prompt 中指定）
- 后续 Agent 自主认领：TaskList → 按优先级排序 → 检查文件冲突 → 无冲突则认领
- 全部有冲突或无可用任务 → 通知 Lead

## 沟通与反馈

### 沟通模式
- **默认直接消息**：CTO→Dev、Dev→Tester 等直接沟通
- **经 Lead 中转**：仅优先级调整、资源冲突、创始人反馈传达
- **广播极少使用**（成本 N 倍）

### 反馈闭环
1. 发现问题 → 发送反馈（位置 + 期望 + 实际 + 建议）
2. Owner 回复：接受 / 拒绝（附理由）
3. 接受 → Owner 更新文档 → 通知"已更新"
4. 反馈方验证 → 闭环

**谁的文档谁更新。未闭环的反馈不允许流程继续推进。**

### 连锁更新
谁改了文档，谁判断是否有连锁影响 → 有影响立即通知受影响文档的 Owner。不允许"先记着后面再改"。

## 运营规则

### 版本管理
- 版本 = 一批功能的统一交付，不是每次 push
- 创始人决定版本边界
- 当前 v0.0.X，正式发布后 v1.0.0
- decisions.md 追加写入，不修改历史

### 代码提交
- 单次 commit 上限 **500 行**（不含 docs/、.pen）
- 超过 500 行拆分为多个可编译的 commit
- 每个 commit 聚焦单一目的

### Token 优化

| 策略 | 做法 |
|------|------|
| 传路径不传内容 | Agent 间传文件路径 |
| Agent 用完即关 | 不保持常驻 |
| 模型分级 | Opus: Designer/CTO, Sonnet: Dev/Tester |

### 文档管理
详见 [`docs/doc-management.md`](./doc-management.md)。

## Context 管理

- 任务切换 → `/clear` 重置 context
- 长任务阶段完成 → `/compact` 压缩保留关键信息
- 大量文件探索 → 用 Explore subagent 隔离，不在主线程读
- Agent 返回大量输出 → 要求结构化摘要，不要全文
- 定期 `/context` 检查 token 消耗

## 会话生命周期

### 开始
1. git pull
2. 读 board.md — 任务全景
3. 读 handoff.md — 上次上下文
4. 读 memory/ — 相关经验
5. 创建本次会话 Tasks

### 结束
1. 关闭所有活跃 Agent
2. 更新 board.md + handoff.md
3. 更新 memory/（如有新经验）
4. git commit + push

### handoff.md 标准结构

```markdown
# Session Handoff — {date}

## 本次完成
- [具体产出]

## 尝试但未成功
- [什么方案、为什么失败]

## 当前阻塞
- [等什么、谁负责]

## 下次启动后立即做
1. [第一步]
2. [第二步]
```
